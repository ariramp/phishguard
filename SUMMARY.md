# PhishGuard - Краткое описание проекта

## 1. Цель проекта

PhishGuard — это гибридная система обнаружения фишинга в электронной почте.  
Система периодически проверяет подключенные IMAP-ящики, извлекает ссылки из новых писем, оценивает риск с помощью ML-сервиса, сохраняет результаты в PostgreSQL и предоставляет операторский веб-интерфейс.

## 2. Архитектура

Компоненты системы:

- `backend` (Go):
  - воркер периодического IMAP-опроса
  - REST API
  - раздача веб-интерфейса
  - бизнес-логика по вынесению вердикта и действиям над письмом
- `ml` (Python/FastAPI):
  - пайплайн оценки по URL + тексту + fusion-слою
  - загрузка артефактов моделей и API предсказаний
- `db` (PostgreSQL):
  - хранение аккаунтов, писем, извлеченных URL, результатов сканирования и ошибок службы

Общий сценарий работы:

1. Через UI/API добавляется IMAP-аккаунт.
2. Воркер запускает опрос ящика с заданным интервалом.
3. Из новых писем извлекаются и нормализуются URL.
4. Backend отправляет URL и контекст письма в ML-сервис.
5. ML возвращает `score/risk/features`.
6. Backend преобразует ответ в итоговый `safe/suspicious/phishing`.
7. Результаты сохраняются в БД и отображаются в интерфейсе и отчётах.
8. При высоком риске применяется действие IMAP (`MOVE/TAG/COPY/NONE`).

## 3. ML-пайплайн (текущая реализация)

В текущей рабочей версии используется трёхуровневая гибридная схема:

- URL-слой:
  - анализ структурных признаков ссылки и домена
  - результат: `url_score`
- Text-слой:
  - анализ `subject + snippet`
  - результат: `text_score`
- Fusion-слой:
  - объединение URL и текстовых признаков
  - результат: `final_score` и `risk`

Текущее правило принятия решения:

- финальный вердикт в backend использует консервативный URL-first подход;
- `phishing` присваивается только при сильных признаках именно URL;
- кейсы с тревожным текстом, но слабым URL-сигналом, переводятся в `suspicious`;
- это снижает число ложноположительных срабатываний на легитимные ссылки.

## 4. Реализованный функционал backend

- Управление IMAP-аккаунтами:
  - создание/просмотр/обновление/удаление
  - включение/пауза аккаунта
  - сброс `last_uid`
- Опрос и сканирование:
  - периодический polling
  - ручной запуск сканирования (`poll once`)
  - ручная проверка отдельной ссылки
- История и аналитика:
  - история, сгруппированная по письму
  - раскрытие деталей по URL внутри письма
  - статистика и временные ряды
  - экспорт обнаружений в CSV
  - сводный отчёт (JSON/CSV)
- Диагностика службы:
  - `healthz`
  - системный статус (`backend/db/ml`)
  - журнал ошибок аккаунтов
- Пересчёт:
  - endpoint для пересчёта уже сохранённых URL по актуальной логике и моделям

## 5. Реализованный функционал интерфейса

Веб-консоль включает:

- карточки общей статистики
- блок сводного отчёта
- график динамики по периодам
- форму подключения/редактирования аккаунта
- действия для аккаунта (`Pause/Resume/Edit/Delete`)
- форму ручной проверки URL
- блок последних ошибок службы
- историю с:
  - фильтром по вердикту
  - поиском
  - раскрытием URL-деталей
- панель действий:
  - обновить данные
  - сканировать сейчас
  - пересчитать историю
  - экспорт CSV-отчётов

## 7. Текущий статус проекта

Проект находится в рабочей стадии MVP+:

- end-to-end сканирование работает;
- интеграция ML работает;
- интерфейс оператора работает;
- отчёты и диагностика работают.

Практические ограничения:

- качество зависит от датасетов и калибровки порогов;
- после обновления логики/моделей рекомендуется запускать пересчёт истории.

## 8. Быстрый запуск

Из корня проекта:

```powershell
docker compose up --build
```

Веб-интерфейс:

- `http://localhost:8080/`

Информация о состоянии ML-модели:

- `http://localhost:8000/v1/model`
